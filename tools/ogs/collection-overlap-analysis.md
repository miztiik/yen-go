## 8. Design Decision: Selected Approach

**Selected**: Premier + Curated tiers only (top 969 collections, ~80% puzzle coverage).

### Rationale

1. **80% coverage at natural tier boundary** — premier + curated = 969 collections, 46,711 puzzles. Clean categorical cutoff, not arbitrary.
2. **Duplication is negligible** — only 69 duplicated puzzles within these tiers. Copying all files per-collection adds <70 extra files.
3. **No need to separate premier/curated directories** — the pipeline's `YL` tag on each SGF identifies collection membership, and `config/collections.json` aliases map OGS names to editorial slugs. Tier information flows from the JSONL metadata into the manifest.
4. **JSONL is the source of truth** — collection identity comes from the JSONL file (OGS collection ID + name), not from `config/collections.json` slugs. The config slugs are the downstream editorial grouping.
5. **Bottom 20% is noise** — community + unvetted tiers contain 2,261 collections averaging 5 puzzles each. Many are user homework with 1–3 puzzles. Not worth the directory noise.

### Non-Latin Collection Names

375 of 969 collections (38.7%) contain non-ASCII scripts. Of these:
- **221 CJK-bilingual** — English portion produces usable slugs automatically
- **43 Latin-extended** — accented chars (French, Spanish, Vietnamese) — slugify handles these
- **111 pure non-Latin** — Russian (63), Thai (50), CJK-only (25), Korean (4), Japanese (3) → produce empty/unusable slugs

**Solution**: `collection-name-translations.json` contains 144 English slug overrides for all collections with bad auto-slugs. The utility loads this as a pre-bootstrap lookup.

### Directory Naming Convention

`{collection_id}-{english_slug}` where:
- `{collection_id}` = OGS collection ID (numeric, from JSONL)
- `{english_slug}` = ASCII-safe slug from collection name, with translation override for non-Latin names (from `collection-name-translations.json`), truncated to 60 chars

## 9. Recommendations (Original Options for Reference)

### Option 1: Collection Manifests + Shared Pool

```
ogs/
├── sgf/                          # Shared physical files (no change)
│   ├── batch-001/
│   │   ├── 2.sgf
│   │   └── ...
│   └── batch-052/
├── collections/
│   ├── index.json                # Master collection list
│   ├── cho-chikun-elementary/
│   │   └── manifest.json         # {puzzles: ["batch-001/10600.sgf", ...]}
│   └── ...
```

**Pros**: Zero duplication, full collection metadata preserved, easy to filter by tier.
**Cons**: Requires a lookup step to find physical files.

### Option 2: Tier-Gated Collection Copy (SELECTED)

1. **Keep only premier + curated** tiers (top 969 collections, ~80% coverage)
2. **Copy SGF files into per-collection directories** with manifests
3. Store under `external-sources/ogs/sgf-by-collection/`

**Pros**: Self-contained, browsable collections. Each dir has its files + metadata.
**Cons**: ~69 duplicated files across tiers (negligible).

### Option 3: Hybrid — Physical Collection Dirs + Dedup via Content Hash

1. Store files physically under `collections/{slug}/` directories
2. At publish time, dedup via content hashing (existing YENGO GN pipeline)
3. The publish pipeline already handles dedup — duplication at source is harmless

**Pros**: Simplest mental model, pipeline handles the hard part.
**Cons**: ~146 extra files on disk (storage cost is minimal for SGF).

## 10. Verdict & Bottom Line

With **1.003x** duplication ratio and only **0.3%** of puzzles appearing in multiple collections, overlap is a **non-issue**.

The real value is in **pruning the long tail**: 2,261 tiny/low-quality collections (community + unvetted) contribute only 11,405 puzzles (19.6%) but create 70% of the directory noise.

**Decision**: Copy premier + curated collections (969 dirs, ~46.7K files) into `sgf-by-collection/` with per-collection manifests. The pipeline's existing dedup handles any residual overlap at publish time.
